💡 Tại sao cần Superpowers?

AI agent không có kỷ luật — nó sẽ nhảy thẳng vào viết code mà không suy nghĩ. Superpowers đặt ra guardrails bắt buộc.

😤
Không có Superpowers
Bạn: "Build me a CLI tool"
AI: Viết code ngay lập tức
Bạn: Nhận ra giữa chừng không đúng ý
Kết quả: Undo hàng giờ công việc
🚀
Có Superpowers
Bạn: "Build me a CLI tool"
AI: Hỏi requirements → Propose design → Bạn approve
AI: Lập kế hoạch chi tiết → Bạn approve
AI: Implement từng task + TDD
  • Brainstorm trước khi code — agent hỏi clarifying questions, propose 2-3 approaches, present design cho bạn approve từng phần
  • Plan trước khi implement — mỗi task 2-5 phút, có file paths cụ thể, complete code, verification steps
  • TDD bắt buộc — viết failing test TRƯỚC, xem fail, viết minimal code, xem pass, commit. Không skip được
  • Subagent review — mỗi task có 2-stage review: spec compliance và code quality, fresh subagent mỗi lần
  • Skills tự động trigger — agent không cần được nhắc. Thấy "let's build X" → brainstorming tự kích hoạt
  • YAGNI + DRY — không build thứ chưa cần, không lặp code. Enforced by skills
Cơ chế

⚙️ Cách Superpowers hoạt động

Superpowers là tập hợp SKILL.md files — mỗi file định nghĩa khi nào kích hoạt và phải làm gì. Agent đọc skill file và thực hiện chính xác theo hướng dẫn.

Session start
using-superpowers
→
Agent scan skills
check triggers
→
Skill match
read SKILL.md
→
Announce
"Using [skill]"
→
Execute
follow exactly
→
Next skill
chain triggers

🔑 Nguyên tắc cốt lõi

  • "If there's a 1% chance a skill applies, you MUST use it" — không rationalize để bỏ qua skill
  • Skills override default behavior nhưng user instructions (AGENTS.md, direct requests) luôn cao hơn
  • Rigid skills (TDD, debugging): follow exactly, không adapt. Flexible skills (patterns): adapt principles to context
  • Instructions say WHAT, not HOW — skill nói mục tiêu, không nói implementation details
  • Context isolation — subagents chỉ nhận context cần thiết cho task đó, không phải toàn bộ session history
💡 Điểm khác biệt với Rules thông thường
AntiGravity Rules luôn active ngầm (passive). Superpowers Skills được invoke on-demand khi trigger condition match — agent chủ động đọc và thực hiện SKILL.md file. Powerful hơn vì mỗi skill có full workflow instructions, không chỉ constraints.

📋 Chi tiết quan trọng

Skill · Before Implementation

🌿 using-git-worktrees — Isolated workspace

Trigger: Sau khi design được approve, trước khi implement. Required cho cả subagent-driven-development và executing-plans.

🌿
Tạo isolated workspace
Agent tạo git worktree trên branch mới — implementation xảy ra ở đây, không phải trên main. Bảo vệ codebase khỏi half-finished work.
🧪
Verify test baseline
Chạy toàn bộ test suite trên worktree mới. Xác nhận tests pass sạch TRƯỚC khi bắt đầu implement. Catch regressions sớm nhất.
Agent thực hiện tự động 
# Agent chạy tự động sau design approval
git worktree add ../my-project-url-shortener feature/url-shortener
cd ../my-project-url-shortener

# Setup project
npm install

# Verify clean baseline
npm test
# → All 47 tests passing. Clean baseline confirmed.
⚠️ AntiGravity limitation
AntiGravity không hỗ trợ parallel subagent execution như Claude Code. Worktree vẫn được tạo nhưng execution là sequential, không parallel. Skills adjust accordingly.
Skill · Step 2

📋 writing-plans — Kế hoạch đủ rõ để junior follow

Trigger: Sau khi spec document được approve. Tiêu chuẩn: Plan phải rõ đến mức "một junior engineer hăng hái nhưng thiếu taste, thiếu judgment, thiếu context project" có thể follow được.

📐 Format một task trong plan

Markdown · docs/plans/2026-03-16-url-shortener.md 
# URL Shortener Implementation Plan
> For Agent: REQUIRED SUB-SKILL: Use superpowers:executing-plans task-by-task

## Task 1: Setup database schema
**Time estimate:** 3 minutes
**File:** src/db/schema.ts

Create SQLite schema with:
- urls table: id (text PK), original_url (text), short_code (text unique),
  created_at (integer), click_count (integer default 0)

```typescript
// Complete code to write:
export const schema = `
  CREATE TABLE IF NOT EXISTS urls (
    id TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(8)))),
    original_url TEXT NOT NULL,
    short_code TEXT UNIQUE NOT NULL,
    created_at INTEGER DEFAULT (unixepoch()),
    click_count INTEGER DEFAULT 0
  );
  CREATE INDEX IF NOT EXISTS idx_short_code ON urls(short_code);
`;
```

**Verification steps:**
1. Run: `node -e "require('./src/db/schema')"` — no errors
2. Run: `npm test -- --testPathPattern=schema` — passes
  • Mỗi task 2-5 phút — không task nào dài hơn. Chia nhỏ nếu cần
  • Phải có exact file paths — không "create a file for X", phải là src/db/schema.ts
  • Phải có complete code hoặc ít nhất interface/signature đầy đủ
  • Phải có verification steps — cụ thể command chạy và expected output
  • Saved tại docs/plans/YYYY-MM-DD-feature-name.md và commit

🔑 2-stage review flow

1️⃣
Spec Compliance Review
Subagent mới kiểm tra: implementation có đúng spec không? Thiếu gì? Thừa gì không cần? Dispatch lại nếu fail.
2️⃣
Code Quality Review
Subagent thứ 3 kiểm tra: code clean? Test coverage? Error handling? Edge cases? Commit sau khi pass cả 2.
💡 AntiGravity — executing-plans thay thế
AntiGravity thiếu parallel subagent → dùng single-flow-task-execution (community port) hoặc executing-plans (sequential). Workflow giống hệt, chỉ là sequential thay vì parallel. Mỗi task vẫn có 2-stage review.
⚠️ Context isolation quan trọng
Subagent KHÔNG nhận toàn bộ session history — chỉ nhận task text (~3-5k chars). Điều này giữ context window của parent agent gọn nhẹ: đọc plan 1 lần (~15k tokens) + ~2k mỗi task dispatch = tổng ~33k thay vì 60k+.
Skill · Implementation

🧪 test-driven-development — RED-GREEN-REFACTOR bắt buộc

Trigger: Khi implement bất kỳ feature hoặc fix nào. Rigid skill — follow exactly, không adapt.

Bước Hành động Xác nhận
🔴 RED Viết failing test trước khi có implementation Chạy test, xác nhận fail với đúng lý do
🟢 GREEN Viết minimal code để test pass Chạy test, xác nhận pass. Không thêm gì extra
🔵 REFACTOR Clean up code, không thay đổi behavior Chạy test lại, xác nhận vẫn pass
💾 COMMIT Commit sau mỗi RED-GREEN-REFACTOR cycle Commit message mô tả test case đã pass
❌ Không được phép
Viết implementation trước khi có test. Viết test và implementation cùng lúc. Skip "watch it fail" step. Đây là rigid skill — agent sẽ từ chối nếu bạn yêu cầu skip TDD.
Skill · Debugging

🐛 systematic-debugging — 4-phase root cause

Trigger: Khi gặp bug, unexpected behavior, test failure, hoặc error. Không guess — follow 4 phases.

1️⃣
Root Cause Tracing
Trace từ symptom ngược về source. Đặt câu hỏi: "Tại sao X xảy ra?" liên tục cho đến khi tìm được actual cause.
2️⃣
Defense in Depth
Validate assumptions. Kiểm tra những điều "hiển nhiên". Log state tại từng bước để xác nhận flow.
3️⃣
Condition-based Waiting
Khi có async/timing issues: wait cho condition thỏa mãn, không hardcode sleep/timeout.
4️⃣
Verification
Sau khi fix, chạy toàn bộ test suite. Confirm fix không tạo ra regression ở nơi khác.
Skill · Quality

🔍 requesting-code-review — Checklist-based review

Trigger: Khi cần review code changes. Dùng checklist thay vì subagent dispatch (AntiGravity adaptation).

  • Correctness — Logic đúng? Edge cases handled? Error paths?
  • Security — Injection risks? Auth/authz? Data exposure? Input validation?
  • Performance — N+1 queries? Unnecessary computation? Memory leaks?
  • Test coverage — Happy path, edge cases, error cases có đủ không?
  • YAGNI compliance — Code có implement thứ gì chưa cần không?
  • DRY compliance — Có duplicate logic không? Có thể extract không?
  • Naming & clarity — Tên biến/function có self-documenting không?

Best Practices

⭐ Pro Tips từ community

🤝
Đừng rush brainstorming
Đây là bước đáng đầu tư nhất. Mỗi câu trả lời của bạn định hình toàn bộ implementation. 10 phút brainstorm = tiết kiệm nhiều giờ rework.
📋
Review spec trước khi proceed
Đọc spec doc agent tạo ra. Đây là lần cuối cùng dễ thay đổi direction. Sau khi approve → plan → execute sẽ rất tốn để undo.
🌿
Luôn dùng worktrees
Không implement trực tiếp trên main. Worktree giúp bạn dễ abandon nếu approach sai mà không mess up codebase.
⏱️
Tasks 2-5 phút
Nếu task trong plan dài hơn 5 phút, agent bị drift. Chia nhỏ hơn. Nhiều small verified steps > ít large unverified steps.
🔄
Reload khi agent drift
Trong long session, agent có thể quên skills. Gõ /superpowers-reload để refresh context.
🧠
Đừng nói "skip TDD"
TDD là rigid skill. Agent sẽ push back. Nếu cần skip (prototyping), nói rõ trong AGENTS.md — user instructions override skills.
⭐ Triết lý cốt lõi
Superpowers không làm cho agent viết code nhanh hơn — nó làm cho agent viết code đúng hơn lần đầu tiên. Đầu tư vào brainstorm và planning = tiết kiệm thời gian debug và refactor sau này. "An enthusiastic junior with poor taste" có thể follow → senior engineer sẽ thấy kết quả ổn.

🔧 Troubleshoot

❓
AntiGravity không thấy workflows
Fix: Đảm bảo đúng path .agent/workflows/*.md. Restart AntiGravity để re-read .agent/ files. Dùng /superpowers-reload.
❓
Agent không follow skills
Fix: Kiểm tra .agent/rules/superpowers.md có tồn tại không. Prompt agent: "You have Superpowers. Read using-superpowers skill."
❓
Files không được write vào artifacts/
Fix: Tell agent: "Please follow the superpowers-execute-plan workflow and write artifacts to disk."
❓
Agent bỏ qua brainstorming
Fix: Explicit trigger: "/superpowers-brainstorm" hoặc "Use brainstorming skill first." Check superpowers.md rule có enforce không.
