OVERVIEW

Codex CLI là gì?

Terminal-first coding agent — đọc repo, edit file, chạy test, commit code — hoàn toàn khác với Codex API cũ (2021)

⚡
CODEX CLI
AI Agent ngay trong terminal của bạn
Codex CLI (ra mắt 16/04/2025) là agentic coding assistant viết bằng Rust, chạy trong terminal, tự đọc codebase, đề xuất thay đổi nhiều file, chạy test và commit — có sandbox cô lập và quy trình approval kiểm soát từng bước. ~4 triệu người dùng mỗi tuần (04/2026).
Codex CLI vs Codex API cũ — Không liên quan!
⚠️
Đừng nhầm lẫn: Codex API (2021) là GPT-3 fine-tune dùng cho Copilot đời đầu, đã bị tắt năm 2023. Codex CLI (2025) là sản phẩm hoàn toàn mới — terminal agent viết bằng Rust, open-source Apache-2.0, không liên quan gì đến Codex cũ.

Cấu hình config.toml — File cấu hình chính
~/.codex/config.toml CONFIG 
# Model mặc định

# Approval policy mặc định
approval_policy = "on-request"  # suggest | on-request | never

# Sandbox mode
sandbox_mode = "workspace-write" # read-only | workspace-write | danger-full-access

# Ngôn ngữ giao tiếp

# Web search (dùng cache để tiết kiệm)
web_search = "cached"           # off | cached | live

# MCP Servers (xem chi tiết ở S10)

  args = ["-y", "@modelcontextprotocol/server-github"]

  [mcp_servers.context7]

Thứ tự ưu tiên config (cao → thấp)
1
Flag -c / --config trong lệnh
Override trực tiếp khi gõ lệnh. Ví dụ: codex -c approval_policy="never" "refactor auth module". Ưu tiên cao nhất, chỉ áp dụng cho lần chạy đó.
2
Project config .codex/config.toml
File trong thư mục dự án (trusted projects only). Commit vào repo để share settings với cả team. Càng gần working directory, càng ưu tiên hơn.
3
Profile file ~/.codex/profile-name.config.toml
Chọn bằng --profile tên-profile. Dùng để có preset khác nhau: "strict" cho production, "auto" cho dev nhanh.
4
User config ~/.codex/config.toml
Config mặc định của bạn — áp dụng cho mọi session. Đặt defaults chung ở đây: model, web_search, preferred language.
5
System config (tổ chức quản lý)
Trên máy managed, tổ chức có thể enforce constraints qua requirements.toml. Ví dụ: cấm dùng approval_policy = "never" hoặc sandbox_mode = "danger-full-access".
AGENTS.md — File hướng dẫn dành cho agent
📋 AGENTS.MD LÀ GÌ
Hướng dẫn Codex về codebase của bạn
AGENTS.md là file Markdown trong repo, Codex đọc khi bắt đầu session. Giống README nhưng viết cho AI — mô tả conventions, architecture, quy tắc không được vi phạm.

Đặt ở root repo → áp dụng toàn bộ. Đặt trong subfolder → áp dụng riêng thư mục đó.
⚡ TỰ TẠO AGENTS.MD
Để Codex tự generate từ codebase
Gõ lệnh /init trong session → Codex phân tích codebase và tạo scaffold AGENTS.md phù hợp.

Bạn review, chỉnh sửa theo thực tế, rồi commit. Mọi session sau đều kế thừa context này — không cần giải thích lại từ đầu.
AGENTS.md — Ví dụ cho dự án Next.js AGENTS.MD 
# Project: E-commerce Platform

## Architecture
- Next.js 14 App Router, TypeScript strict mode
- Database: PostgreSQL via Prisma ORM
- Auth: NextAuth.js v5 with JWT
- Styling: Tailwind CSS + shadcn/ui

## Conventions
- All API routes in `src/app/api/` follow REST conventions
- Shared types in `src/types/` — never inline type definitions
- Use `zod` for ALL input validation, never trust raw user input
- Error handling: always use custom `AppError` class in `src/lib/errors.ts`

## Do NOT
- Never commit secrets or API keys
- Never modify `prisma/migrations/` manually
- Never use `any` type — use `unknown` and narrow
- Never skip error boundaries in Server Components

## Testing
- Unit tests: Vitest in `__tests__/` next to source files
- E2E tests: Playwright in `e2e/`
- Run `pnpm test` before every commit
APPROVAL

Approval Modes — Kiểm soát mức tự chủ của Agent

Chọn mức độ Codex được phép làm gì mà không cần xin phép

🔐
APPROVAL POLICY
3 chế độ kiểm soát — bạn quyết định mức độ tin tưởng
Approval policy quyết định khi nào Codex cần hỏi bạn trước khi thực hiện. Kết hợp với Sandbox Mode để tạo nên mức bảo mật phù hợp với từng tình huống.
Ba Approval Modes
💡 SUGGEST
Chế độ đề xuất — Hỏi từng bước
Codex hiển thị diff và chờ bạn approve trước mọi thay đổi file và lệnh shell.

Phù hợp khi: learning, lần đầu dùng với codebase mới, hoặc task có rủi ro cao.

--ask-for-approval suggest
⚡ ON-REQUEST (Auto)
Tự động trong workspace — Hỏi khi ra ngoài
Codex tự edit file và chạy lệnh trong working directory. Chỉ hỏi khi: edit file ngoài workspace, cần network access, hoặc MCP tool destructive.

Recommended cho daily work.

--ask-for-approval on-request
⚠️ NEVER (YOLO)
Không hỏi — Chạy hoàn toàn tự động
Codex không hỏi gì cả — làm hết theo judgment của model. Alias: --yolo hoặc --dangerously-bypass-approvals-and-sandbox.

Chỉ dùng trong môi trường hardened / CI đã được kiểm soát chặt.

--ask-for-approval never
Destructive actions luôn cần approval — Không ngoại lệ
⚠️
Quy tắc bất di bất dịch: Dù bạn dùng approval mode nào, các MCP/app tool call được đánh dấu là destructive (xóa data, gửi email, push code...) luôn luôn cần approval riêng — ngay cả khi tool cùng đó có annotation read-only. Codex không bỏ qua điều này.
Cú pháp sử dụng Approval Modes
Terminal — Approval mode examples CLI 
# Mode suggest: hỏi trước mọi thay đổi (an toàn nhất)
codex --ask-for-approval suggest "refactor the auth module"

# Mode on-request: tự động trong workspace (recommended)
codex --sandbox workspace-write --ask-for-approval on-request \
       "add unit tests to all service files"

# Non-interactive (exec mode) — chạy task rồi thoát
codex exec --sandbox workspace-write \
       "fix all TypeScript errors in src/"

# YOLO mode — chỉ dùng trong CI/CD hardened
codex --yolo "bump all dependencies to latest"

# Switch sang read-only mid-session
/permissions   # trong TUI → chọn read-only mode

# Lưu approval policy vào config.toml thay vì gõ mỗi lần
approval_policy = "on-request"   # trong ~/.codex/config.toml
SANDBOX

Sandbox Modes — Ranh giới bảo vệ hệ thống

Sandbox giới hạn những gì Codex có thể chạm tới — bất kể approval mode

Ba Sandbox Modes
👁️ READ-ONLY
Chỉ đọc — Không thay đổi gì
Codex có thể đọc file, phân tích code, generate suggestions — nhưng không thể ghi bất cứ file nào, không chạy lệnh có tác động.

Dùng khi: code review, audit, explain codebase cho người mới.
✏️ WORKSPACE-WRITE
Ghi trong workspace — Default recommended
Codex tự do đọc/ghi file trong working directory. Network access tắt mặc định. Để ra ngoài workspace hoặc dùng network → hỏi approval.

Default cho daily work. Cân bằng tốt giữa tốc độ và an toàn.
🔓 DANGER-FULL-ACCESS
Toàn quyền hệ thống
Không có giới hạn — Codex có thể làm mọi thứ trên hệ thống của bạn bao gồm network, system files, external commands.

Chỉ dùng trong Docker container hoặc VM cô lập hoàn toàn.
Bật Network Access trong workspace-write
~/.codex/config.toml — Bật network cho domain cụ thể SANDBOX CONFIG 
sandbox_mode = "workspace-write"

# Cho phép Codex gọi API cụ thể (trong workspace-write)
[sandbox.network]
allow_domains = [
  "api.github.com",
  "registry.npmjs.org",
  "pypi.org",
]

# Hoặc bật full network (cẩn thận)
# allow_full_network = true

# Cho Codex Cloud: bật internet access
# Vào agent settings → Internet access → enable
# hoặc cấu hình domain allow list
REMOTE

Remote Control — Điều khiển Codex từ xa

Kết nối TUI từ máy khác, giao task qua Telegram/Discord, hoặc expose như WebSocket server

📡
APP-SERVER
Codex như một daemon — luôn sẵn sàng nhận lệnh
Dùng codex app-server để khởi động Codex như background service lắng nghe WebSocket. Kết nối từ bất kỳ TUI nào, từ plugin Telegram, Discord, hay CI script — không cần session tương tác.
Ba cách dùng Remote Control
🖥️ REMOTE TUI
Attach TUI vào session đang chạy
Khởi động app-server trên máy A, attach TUI từ máy B qua SSH hoặc local. Giống tmux attach nhưng cho Codex session — tiếp tục công việc từ chỗ đã dừng.
📱 TELEGRAM / DISCORD
Giao task qua messaging
Plugin community cho phép send task đến Codex qua Telegram hoặc Discord. Two-way communication qua MCP — nhận progress update và kết quả ngay trên phone.
🔌 CI/CD INTEGRATION
Nhúng vào pipeline
codex exec chạy non-interactive, output JSON events, exit code theo success/fail — tích hợp vào GitHub Actions, GitLab CI, Jenkins như bất kỳ CLI tool nào.
Thiết lập App Server
Terminal — App Server & Remote Connection REMOTE 
── Trên máy chủ (server / laptop bạn để chạy) ──

# Khởi động app-server (foreground)
codex app-server --listen ws://0.0.0.0:8765

# Hoặc chạy như daemon (background)
codex app-server --daemonize --listen ws://localhost:8765

# Kiểm tra daemon đang chạy
codex app-server --status

── Từ máy khác / terminal khác ──

# Attach TUI vào app-server qua WebSocket
codex --remote ws://server-ip:8765

# Hoặc qua WSS (secure)
codex --remote wss://server-ip:8765

── Non-interactive exec vào remote server ──

codex exec --remote ws://server-ip:8765 \
    "run all failing tests and fix them"

── Resume session đã có (không mất context) ──

codex --resume                    # resume gần nhất
codex --resume <session-id>      # resume session cụ thể
codex /agent                     # trong TUI: chọn thread từ picker

MULTI-AGENT

Multi-Agent — Nhiều agent song song

Delegate task phức tạp cho subagent chuyên biệt — chạy song song, tăng tốc đáng kể

Cách Multi-Agent hoạt động

Root agent nhận task → phân tích → spawn subagents cho các phần độc lập → collect results → synthesize. Mỗi subagent có context window riêng, không bị ảnh hưởng lẫn nhau.

🌳 ROOT AGENT
Orchestrator — Phân phối và tổng hợp
Nhận task ban đầu, lập kế hoạch phân chia, spawn subagents với context cụ thể, theo dõi progress, tổng hợp kết quả cuối cùng. Bạn chỉ giao task cho root agent.
🔧 SUBAGENTS
Specialists — Làm sâu trong domain riêng
Mỗi subagent có identity riêng (subagent_identity trong hook inputs), làm việc trong scope hẹp, báo cáo lại root. Cấu hình thread caps và wait-time để tránh over-spawning.
Ví dụ thực tế: PR Review với Multi-Agent
🔍 Task: "Review toàn bộ PR #142 trước khi merge"
R
Root agent đọc diff của PR, phân chia thành 4 domains: security, performance, tests, style
A1
Agent 1 — Security review: kiểm tra injection, auth bypass, secrets exposure
A2
Agent 2 — Performance: N+1 queries, memory leaks, unoptimized loops
A3
Agent 3 — Test coverage: missing unit tests, edge cases, mock completeness
A4
Agent 4 — Code style: naming conventions, AGENTS.md compliance, dead code
R
Root agent tổng hợp → tạo review comment GitHub với severity ranking → hỏi approval trước khi post
Cấu hình Multi-Agent trong config.toml
.codex/config.toml — Multi-Agent settings CONFIG 
[multi_agent]
# Số subagents tối đa cùng lúc (tránh rate limit)
thread_cap = 4

# Thời gian chờ subagent respond trước khi timeout
wait_time_seconds = 120

# Root agent hints — Codex biết đây là orchestrator
root_agent_hint = true

# Depth tối đa của subagent tree
max_depth = 3

SKILLS

Skills — Workflow có thể tái sử dụng

Viết SKILL.md một lần, Codex tự động nhận diện và áp dụng khi task phù hợp

📖
SKILLS
Codex học workflow của bạn một lần, dùng mãi mãi
Skills là thư mục chứa SKILL.md + scripts tuỳ chọn. Codex load metadata (tên, mô tả) cho mọi skill, và chỉ đọc full instructions khi quyết định dùng — tiết kiệm context window. Skills có thể invoke explicitly hoặc Codex chọn tự động khi task match.
Vị trí đặt Skills
🌐 GLOBAL SKILLS
~/.agents/skills/ — Dùng trên mọi project
Skills đặt ở thư mục home — available trong mọi session, mọi repo. Dùng cho workflow cá nhân lặp đi lặp lại: deploy checklist, commit message format, code review style…
📁 PROJECT SKILLS
.agents/skills/ — Riêng cho project
Skills trong repo — commit vào version control, share với cả team. Dùng cho workflow đặc thù của dự án: API boilerplate, migration scripts, test generation patterns…
Tạo SKILL.md — Cấu trúc đầy đủ
~/.agents/skills/deploy-checklist/SKILL.md SKILL 
---
name: deploy-checklist
description: Pre-deployment validation workflow. Triggers when user mentions
  "deploy", "release", "production", or "ship to prod". Runs test suite,
  checks env variables, validates build, creates release notes.
---

## Deploy Checklist Workflow

Khi được gọi, thực hiện theo thứ tự sau:

1. Chạy `pnpm test` — nếu fail, dừng lại và report lỗi
2. Chạy `pnpm build` — kiểm tra không có TypeScript error
3. Verify tất cả env vars trong `.env.example` đã có trong `.env.production`
4. Chạy `pnpm run lint` — không được có error (warning ok)
5. Kiểm tra CHANGELOG.md đã được update cho version mới
6. Tạo git tag theo format `v{major}.{minor}.{patch}`
7. Hỏi confirmation trước khi push tag và trigger CI

## Abort conditions
- Bất kỳ test nào fail → dừng hẳn, không tiếp tục
- Missing env var → list ra tất cả missing, hỏi user điền trước
- Build error → hiện full error, suggest fix

MCP

MCP Servers — Custom tools và context

Model Context Protocol — TOML config, STDIO hoặc HTTP, tự động launch khi session bắt đầu

Hai loại MCP server trong Codex
🖥️ STDIO (LOCAL)
Chạy như local process
Codex launch server như child process, giao tiếp qua stdin/stdout. Dành cho tools cài local: npm packages, uvx (Python), binary tools.

Codex tự khởi động khi session bắt đầu — không cần chạy tay.
🌐 STREAMABLE HTTP (REMOTE)
Server ở URL cố định
Codex connect đến URL — dành cho cloud services, shared team servers, OAuth-protected APIs.

Hỗ trợ OAuth login/logout trực tiếp từ Codex với codex mcp add --auth oauth URL.
Cấu hình MCP servers trong config.toml
~/.codex/config.toml — MCP Server config MCP CONFIG 
# ── STDIO servers (local process) ──

[mcp_servers.github]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]
env = { GITHUB_TOKEN = "ghp_xxxx" }   # least privilege!

[mcp_servers.postgres]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-postgres",
         "postgresql://readonly_user:pass@localhost/mydb"]

[mcp_servers.playwright]
command = "npx"
args = ["-y", "@playwright/mcp@latest"]

[mcp_servers.openai-docs]
command = "npx"
args = ["-y", "@openai/mcp-server-docs"]

# ── Streamable HTTP servers ──

[mcp_servers.context7]
type = "streamable-http"
url = "https://mcp.context7.com/mcp"

[mcp_servers.figma]
type = "streamable-http"
url = "https://api.figma.com/mcp"
auth = "oauth"  # Codex tự mở browser để login
