Kiro — IDE lên kế hoạch trước khi code

Kiro là agentic IDE do AWS xây dựng, ra mắt giữa 2025, chạy trên nền VS Code và được powered by Claude Sonnet 4.5 qua Amazon Bedrock. Điểm khác biệt cốt lõi: thay vì nhận prompt → sinh code ngay, Kiro bắt bạn đi qua một quy trình có cấu trúc — tạo tài liệu spec, review, rồi mới code.

Các bước cài đặt
1

Download Kiro IDE

Truy cập kiro.dev → chọn platform (macOS/Windows/Linux) → download bộ cài. File cài khoảng 200-300MB.

2

Đăng nhập AWS Builder ID

Khi mở lần đầu, Kiro yêu cầu đăng nhập AWS Builder ID (miễn phí, khác với tài khoản AWS thông thường). Đăng ký tại builder.aws.com nếu chưa có.

3

Mở Project

File → Open Folder, hoặc chạy kiro . từ terminal trong thư mục project. Kiro sẽ tự detect stack và hỏi bạn có muốn generate steering docs không.

4

Generate Steering Docs lần đầu

Click panel Kiro (icon 👻 ở sidebar) → Generate Steering Docs. Agent sẽ scan repo và tự tạo 3 file trong .kiro/steering/: product.md, tech.md, structure.md.

5

Thêm .kiro/ vào .gitignore đúng cách

Chỉ ignore cache và sessions, không ignore specs và steering — đây là "source of truth" của team cần commit.

.gitignoreGIT

# Kiro — NÊN commit (specs, steering, hooks)
# .kiro/specs/        ← để lại, commit vào Git

# .kiro/hooks/        ← để lại, commit vào Git

SPECS

Spec-Driven Development

Ba file markdown, quy trình plan-first, ví dụ tạo tính năng đăng nhập

Spec là gì — Cấu trúc 3 file

Mỗi tính năng lớn trong Kiro là 1 spec — một folder chứa 3 file markdown. Kiro tạo ra chúng từ prompt của bạn, bạn review và approve, rồi AI mới code:

.kiro/ specs/ user-authentication/ requirements.md ← Yêu cầu nghiệp vụ (viết bằng EARS notation) design.md ← Kiến trúc kỹ thuật, component diagram tasks.md ← Danh sách task cụ thể, AI tick khi xong product-review-system/ payment-integration/
Quy trình tạo Spec — Ví dụ: Tính năng Review sản phẩm
💡
Cách tạo spec: Trong Kiro Chat, gõ Create Spec: [mô tả tính năng]. Kiro sẽ đọc steering files, scan codebase, rồi tạo ra 3 file markdown để bạn review trước khi viết code.
BẠN

KIRO 👻
Đang đọc steering files của project... Scan workspace... Tạo spec product-review-system:

✅ requirements.md — 8 yêu cầu nghiệp vụ (EARS notation)
✅ design.md — Component diagram, DB schema, API endpoints
✅ tasks.md — 12 implementation tasks

Hãy review 3 file này trong .kiro/specs/product-review-system/ trước khi tôi bắt đầu code nhé?
.kiro/specs/product-review-system/requirements.mdREQUIREMENTS
# Product Review System — Requirements

## Functional Requirements

# EARS Notation: WHEN [trigger] THE SYSTEM SHALL [behavior]

REQ-001: WHEN an authenticated user views a product page
         THE SYSTEM SHALL display a "Write a Review" button.

REQ-002: WHEN a user submits a review
         THE SYSTEM SHALL validate that rating is between 1-5
         AND text is between 10-1000 characters.

REQ-003: WHEN a product has reviews
         THE SYSTEM SHALL display average rating rounded to 1 decimal
         AND total review count on the product page.

REQ-004: WHEN a user has already reviewed a product
         THE SYSTEM SHALL allow editing but NOT duplicate reviews.

# Tech Stack & Conventions

## Language & Runtime
- TypeScript 5.x với strict mode bật
- Node.js 20 LTS, sử dụng ESM modules
- Package manager: pnpm (KHÔNG dùng npm hay yarn)

## Framework
- Backend: Hono.js (KHÔNG dùng Express)
- Frontend: React 19 + Vite
- Database: PostgreSQL với Drizzle ORM

## Coding Standards
- Tên biến/function: camelCase
- Tên type/interface: PascalCase
- Error handling: LUÔN dùng Result<T, E> type, KHÔNG throw
- API responses: { data, error, meta } format

## Testing
- Unit tests: Vitest
- Mọi repository method đều phải có test
- Coverage target: 80%

# Security Rules — LUÔN áp dụng

NEVER hardcode secrets, API keys, hoặc credentials vào code.
NEVER log sensitive user data (email, phone, password).
ALWAYS validate và sanitize input trước khi query DB.
ALWAYS dùng parameterized queries, KHÔNG string concatenation.
ALWAYS check authentication trước khi trả về data của user.

Agent Hooks — Automation trong IDE

"If this, then that" cho developer workflow — chạy tự động không cần prompt

Hooks là gì — Tại sao cần?

Agent Hooks là các automated triggers — khi có sự kiện xảy ra trong IDE (lưu file, tạo file, xoá file...), Kiro tự động chạy một action AI đã được định nghĩa trước, không cần bạn prompt thủ công.

🪝 Không có Hooks
Quy trình thủ công
Lưu file → nhớ mở Kiro Chat → gõ "write tests for this file" → chờ → copy kết quả → tạo test file. Mỗi lần làm mới đều lặp lại.
⚡ Có Hooks
Tự động hoàn toàn
Lưu file → Hook tự trigger → Kiro tự tạo test file bên cạnh, tự chạy linter, tự update docs. Bạn chỉ việc code.
Cấu trúc Hook File

Hooks sống trong .kiro/hooks/ dưới dạng JSON file — được commit vào Git để cả team dùng chung:

.kiro/hooks/auto-test-generator.jsonHOOK
{
  "name": "Auto Test Generator",
  "description": "Tự tạo test file khi lưu React component",
  "enabled": true,
  "trigger": {
    "event": "fileSave",            // sự kiện: fileSave | fileCreate | fileDelete | manual
    "filePattern": "src/components/**/*.tsx"  // chỉ trigger cho file này
  },
  "action": {
    "type": "agent",
    "prompt": "Xem file React component vừa lưu. Nếu chưa có file test tương ứng ở __tests__/, hãy tạo file test với các test cases cover: render không crash, props cơ bản, và các state changes quan trọng. Dùng Vitest + React Testing Library."
  }
}

BẠN
Xem schema của bảng users trong DB, rồi tạo cho tôi TypeScript types tương ứng với Zod validation schema.
KIRO 👻 + PostgreSQL MCP
[Đang query DB qua MCP...]

Tôi thấy bảng users có: id (uuid), email (varchar 255, unique), name (varchar 100), created_at (timestamptz), role (enum: 'user','admin'), avatar_url (varchar nullable).

Đây là TypeScript types + Zod schema:
src/types/user.ts — Generated from live DB schemaAUTO-GENERATED
import { z } from "zod";

export const UserRole = z.enum(["user", "admin"]);

export const UserSchema = z.object({
  id: z.string().uuid(),
  email: z.string().email().max(255),
  name: z.string().min(1).max(100),
  createdAt: z.string().datetime(),
  role: UserRole.default("user"),
  avatarUrl: z.string().url().nullable(),
});

export type User = z.infer<typeof UserSchema>;
export type UserRole = z.infer<typeof UserRole>;

Use Case 5 — Quality Gate Pipeline tự động

Chain nhiều hooks lại để tạo CI/CD-like pipeline ngay trong IDE — mỗi khi lưu file:

Quality Gate: save → lint → test → security → docsHOOK CHAIN
File saved trigger:
  ✅ Hook 1: ESLint check → auto-fix fixable issues
  ✅ Hook 2: Nếu có function mới → generate Vitest unit tests
  ✅ Hook 3: Security scan → flag nếu có hardcoded data
  ✅ Hook 4: Nếu là API route → update OpenAPI spec

Kết quả: Mỗi lần Ctrl+S, file của bạn đã được:
  → Clean code (lint)
  → Có tests (auto-generated)
  → Secure (scan)
  → Documented (docs update)
Tips nâng cao
💡
Đầu tư vào Steering trước
Thời gian viết steering files tốt (30-60 phút đầu project) sẽ tiết kiệm hàng giờ giải thích sau này. Kiro output tốt đến đâu phụ thuộc vào steering tốt đến đó.
📋
Review Spec trước khi code
Đừng bỏ qua bước review requirements.md và design.md. Sửa spec mất 5 phút. Sửa code đã viết sai spec mất 5 giờ.
✅
Commit .kiro/ vào Git
Specs + hooks + steering là "source of truth" của team. Version control chúng như code thật để cả team dùng chung và review được.
⚠️
Kiro không thay thế engineering judgment
Kiro là accelerator cho developer có kỷ luật, không phải magic wand. Bạn vẫn cần hiểu code được tạo ra để maintain nó.
